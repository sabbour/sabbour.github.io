<!--kg-card-begin: markdown--><h1 id="introduction">Introduction</h1>
<p>In this blog post, I'll try going over the most common reasons for performance issues you may run into when you deploy PHP websites on Azure App Services, as well as tips and tricks on how to optimize them.</p>
<p>At the end of the article, I'll show you how a fairly simple CodeIgniter website behaves before and after the optimizations, through a series of load tests to determine the breaking point of each configuration.</p>
<h1 id="butitisrunningonthecloudwhydoineedtooptimize">But it is running on the cloud, why do I need to optimize?!</h1>
<p>Although Azure App Services are scalable, there are indeed some core differences in the way they are designed in order to give you the instant cloud scalability that the cloud is all about. When moving from a monolithic architecture of hosting on a single stand alone server to the cloud, these differences force you to rethink about the way you do things that on the longer run, will enable your application to scale in a much better way.</p>
<h2 id="whatsortofdifferences">What sort of differences?</h2>
<p>In order to give you that instant scalability with no downtime, Azure App Services instances (managed Virtual Machines) have a shared network storage between them. This storage allows them to share the application code and content. Where you may face issues here depends on how your application accesses the file system.</p>
<p>By default, to maintain session state, the load balancer that is in-front of your Azure App Service instances uses sticky sessions through the use of Application Request Routing (ARR) Session Affinity cookies, to maintain client connections with a single server.</p>
<h2 id="typicalproblematicpatternstowatchoutforandhowtosolvethem">Typical problematic patterns to watch out for and how to solve them</h2>
<h3 id="heavyfileio">Heavy File I/O</h3>
<p>If your application has heavy IO, typically if you a lot of images/static content that are stored and accessed via regular disk IO operations, during high traffic, these will strain the network file system that all the instances are reading from.</p>
<p>An easy solution here is, that doesn't require rewriting the application code, is to off-load the loading of static content (images, css, js, downloads, ..) to a <a href="https://azure.microsoft.com/en-us/services/cdn/">Content Delivery Network (CDN)</a>.</p>
<p><strong>Using a CDN has many benefits:</strong></p>
<ol>
<li>Improve the end user's experience by caching those files at a much closer location than the Azure datacenter, possibly even in the same city.</li>
<li>Save Azure App Service CPU cycles and file system access as the files will be served directly from the CDN at a much higher throughput</li>
<li>Browsers typically limit the number of concurrent connections to the same domain. So hosting the content on a separate domain (the CDN) will allow the browser to download more content in parallel, hence improving the page load time.</li>
</ol>
<p>Another option, if you have the luxury of modifying your application code, or there exists plugins for it, would be to outsource the storage of those files and content generated photos to Azure Blob Storage. As a bonus, you can enable a CDN on the blob storage too.</p>
<h3 id="filelockingandexcessivelogging">File locking and excessive logging</h3>
<p>The pattern above gets worse if you use file locking mechanisms/functions like <a href="http://php.net/manual/en/function.flock.php">flock() function in PHP</a> before accessing files.</p>
<p>Typical situations where this happen is when you are using some logging libraries and you log very frequently, depending on how many instances are part of your App Service, many of them are now competing on locking and accessing the same file to write their logs, bringing the application to a crawl, especially on websites with high requests per second.</p>
<p>To get around this issue, you may reduce the verbosity of logs in the production environment, outsource logging to an <a href="https://azure.microsoft.com/en-us/services/storage/tables/">Azure Storage Table</a>, use <a href="https://azure.microsoft.com/en-us/services/application-insights/">Azure App Insights</a>, or use unique names per server instance for the log files so that different instances are not competing to write to the same log file.</p>
<h3 id="outgoingportexhaustion">Outgoing port exhaustion</h3>
<p>If you are running an application that does a lot of outgoing connections, for example simply to connect to a MySQL database, you need to be careful and optimize how those connections are handled. You'll know when you run into this issue because the Azure App Service Advisor will display the following &quot;Socket Resources Exhausted&quot; warning on the portal.</p>
<p><img src="__GHOST_URL__/content/images/2016/04/Socket-Resources.JPG" alt="Socket Resources Exhausted" loading="lazy"></p>
<p>This requires more effort to solve, as it usually indicates some in-efficiency in the way the application is accessing remote resources, such as MySQL databases. You will most likely be able to avoid the issue with the use of some sort of <strong>caching layer</strong> between the application and the database, or through the use of <strong>persistent database connections</strong>.</p>
<h3 id="sessionaffinity">Session affinity</h3>
<p>If you don't need session affinity in your deployment, because your website is truly stateless or you've outsources the state to a different mechanism, like Azure Tables or Redis Cache, <a href="https://azure.microsoft.com/en-us/blog/disabling-arrs-instance-affinity-in-windows-azure-web-sites/">disabling Session Affinity</a> will improve performance because then the incoming requests will be fairly load balanced between your different instances.</p>
<p>To do so, you add the below in your web.config file in the system.Webserver area<br>
<code>&lt;system.Webserver&gt;   &lt;httpProtocol&gt;     &lt;customHeaders&gt;        &lt;add name=&quot;Arr-Disable-Session-Affinity&quot; value=&quot;true&quot; /&gt;      &lt;/customHeaders&gt;    &lt;/httpProtocol&gt; &lt;system.Webserver&gt;</code></p>
<h3 id="enablecompressedoutputgzip">Enable compressed output (gzip)</h3>
<p>Compressing output of pages and text based content (js/css) improves page load times for your end users since less data is transferred. It adds a bit of overhead to your server processing though. In general, it is a good idea to enable it.</p>
<p>Depending on your application, this can be either enabled in your application settings (for example by enabling the <strong>compress_output</strong> config of CodeIgniter), or by enabling it in the web.config like the following:</p>
<p><code>&lt;system.webServer&gt;    &lt;urlCompression doStaticCompression=&quot;true&quot; doDynamicCompression=&quot;true&quot; /&gt;    &lt;httpCompression&gt;       &lt;dynamicTypes&gt;          &lt;clear /&gt;          &lt;add enabled=&quot;true&quot; mimeType=&quot;text/*&quot; /&gt;          &lt;add enabled=&quot;true&quot; mimeType=&quot;message/*&quot; /&gt;          &lt;add enabled=&quot;true&quot; mimeType=&quot;application/x-javascript&quot; /&gt;          &lt;add enabled=&quot;true&quot; mimeType=&quot;application/javascript&quot; /&gt;          &lt;add enabled=&quot;true&quot; mimeType=&quot;application/json&quot; /&gt;          &lt;add enabled=&quot;false&quot; mimeType=&quot;*/*&quot; /&gt;          &lt;add enabled=&quot;true&quot; mimeType=&quot;application/atom+xml&quot; /&gt;          &lt;add enabled=&quot;true&quot; mimeType=&quot;application/atom+xml;charset=utf-8&quot; /&gt;       &lt;/dynamicTypes&gt;       &lt;staticTypes&gt;          &lt;clear /&gt;          &lt;add enabled=&quot;true&quot; mimeType=&quot;text/*&quot; /&gt;          &lt;add enabled=&quot;true&quot; mimeType=&quot;message/*&quot; /&gt;          &lt;add enabled=&quot;true&quot; mimeType=&quot;application/javascript&quot; /&gt;          &lt;add enabled=&quot;true&quot; mimeType=&quot;application/atom+xml&quot; /&gt;          &lt;add enabled=&quot;true&quot; mimeType=&quot;application/xaml+xml&quot; /&gt;          &lt;add enabled=&quot;true&quot; mimeType=&quot;application/json&quot; /&gt;          &lt;add enabled=&quot;false&quot; mimeType=&quot;*/*&quot; /&gt;       &lt;/staticTypes&gt;    &lt;/httpCompression&gt; &lt;/system.webServer&gt;</code></p>
<h3 id="outputcaching">Output caching</h3>
<p>Output caching is when you store the rendered HTML from a dynamic page on the server so that the next request is served from the cache and doesn't hit the PHP process. This dramatically improves performance as you're no longer executing the PHP script for every request, and you're not hitting the database for every request either.</p>
<p>Like HTTP Compression, this can be done in several places. If you're using a PHP framework like Code Igniter, you can enable output caching by calling the following method in your controller actions <code>$this-&gt;output-&gt;cache(N);</code> where <strong>N</strong> is the number of minutes to cache the response. Joomla, WordPress and others also have their own cache plugins.</p>
<p>Another place to do so would be on the IIS level. This would give you less control on what to cache, but it is an easy way to enable site-wide caching. The below configuration will cache all requests for 1 minute, and vary the cache key by any query strings. You'll need to add the below in your web.config in the system.Webserver area:<br>
<code>&lt;system.webServer&gt;    &lt;caching&gt;       &lt;profiles&gt;          &lt;add varyByQueryString=&quot;*&quot; extension=&quot;*&quot; policy=&quot;CacheForTimePeriod&quot; kernelCachePolicy=&quot;CacheForTimePeriod&quot; duration=&quot;00:01:00&quot; /&gt;       &lt;/profiles&gt;    &lt;/caching&gt; &lt;/system.webServer&gt;</code></p>
<h2 id="cachecontrolheaders">Cache-control headers</h2>
<p>For static files, you should almost always serve Cache-control headers with the server responses. This allows the clients (browsers) and CDNs to cache those files and avoid requesting them again. The below will set a MaxAge of 7 days to static content. To do so, add the below to your web.config in the system.Webserver area:<br>
<code>&lt;system.webServer&gt;     &lt;staticContent&gt;         &lt;clientCache cacheControlMaxAge=&quot;7.00:00:00&quot; cacheControlMode=&quot;UseMaxAge&quot; /&gt;     &lt;/staticContent&gt; &lt;/system.webServer&gt;</code></p>
