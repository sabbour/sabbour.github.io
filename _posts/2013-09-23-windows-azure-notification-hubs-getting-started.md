<!--kg-card-begin: markdown--><div style="padding: 12px; background-color: rgb(255, 255, 153);">   <p>This is part 1 of a 3 part series where we will get to know how Notification Hubs work and why they are a cool feature of Windows Azure.</p>    <p>You can get the source code of the projects in this blog post here: <a href="http://github.com/sabbour/notificationhubw8">http://github.com/sabbour/notificationhubw8</a></p> </div>  <h1>Push Notifications you say?</h1>  <p>Most users of mobile applications now expect to be notified when something of interest to them happens in the app, instead of having to regularly check. So if you are creating a mobile app, chances are you will need to use push notifications functionality sooner or later.</p>  <p>But real world apps have complex needs, whether you are pushing to <strong>multiple platforms</strong>, requiring <strong>personalization</strong> and <strong>localization</strong> of the pushed messages, handling <strong>different mobile app versions</strong> and most importantly <strong>scale</strong>.</p>  <h1>Challenges of push notifications</h1>  <p>Push notifications are not as trivial to implement as they may sound, an end-to-end solution requires a lot of backend code that you would have to write and maintain.</p>  <h2>Platform dependency</h2>  <p>Each push notification service (PNS) has different protocols (e.g., HTTP vs. TCP, xml payload vs. JSON payload) as well as different presentation formats and capabilities (tiles vs. toasts vs. badges).</p>  <h2>Routing</h2>  <p>A PNS provides you a way to send a message to a certain device, but doesn’t provide a way to do “broadcast” messages that reach multiple devices. If you need to send a message to multiple devices, such as a breaking news notification, or specific interest groups, such as people who are interested in “Sports” you would have to write that in your backend and maintain a registry associating device handles to interest groups/users.</p>  <h2>Scale</h2>  <p>Storing device handles (which expire), and broadcasting to millions of devices with low latency requires parallelization in both, the backend machines and the database, meaning high storage and machine costs.</p>  <h1>Notification Hubs to the rescue!</h1>  <p>Before diving into how Notification Hubs work (and diving into the code :) ) let’s compare how to send a push notification using your own backend vs using the Notification Hubs.</p>  <h2>Push without Notification Hubs</h2>  <p><a href="/content/images/msdn/1185.image_5F00_1DB10669.png"><img title="Push notification lifecycle" style="border: 0px currentcolor; display: inline;" border="0" alt="Push notification lifecycle" src="/content/images/msdn/1526.image_5F00_thumb_5F00_21BDE4EC.png" width="318" height="326" /></a> </p>  <ol>   <li><strong>Client</strong> app requests a channel/unique identifier from the PNS.</li>    <li><strong>PNS</strong> replies with a unique identifier.</li>    <li><strong>Client</strong> sends its unique identifier to your application backend.</li>    <li><strong>Application</strong> backend stores/updates the unique identifier of the client.</li>    <li><strong>Application</strong> constructs a platform specific message, implements PNS protocol, contacts PNS to push a notification to the client.</li> </ol>  <h2></h2>  <h2>Push with Notification Hubs</h2>  <p><a href="/content/images/msdn/6862.image_5F00_6A0B2ABB.png"><img title="image" style="border: 0px currentcolor; display: inline;" border="0" alt="image" src="/content/images/msdn/6811.image_5F00_thumb_5F00_49867BBA.png" width="365" height="351" /></a></p>  <ol>   <li><strong>Client</strong> app requests a channel/unique identifier from the PNS.</li>    <li><strong>PNS</strong> replies with a unique identifier.</li>    <li><strong>Client</strong> sends its unique identifier to the <strong>Notification Hub.</strong></li>    <li><strong>Application </strong>contacts the Notification Hub to send push notifications.</li>    <li><strong>Notification Hub</strong> contacts the PNS to push a notification to the client.</li> </ol>  <p>Now why would the Notification Hubs scenario be better, you might ask:</p>  <ol>   <li>Application backend <strong>does not have to implement any PNS specific code</strong>, it uses the Notification Hub API to push messages.</li>    <li>Application backend <strong>neither stores nor maintains device information</strong>, it is handled by the Notification Hub.</li>    <li>Application can <strong>broadcast to millions of devices with a single call</strong> to the Notification Hub, <strong>instead of looping on the database and pushing one by one.</strong></li> </ol>  <h1>Let’s start!</h1>  <h2>Creating the Notification Hub</h2>  <p>Login to the Windows Azure management portal, click on New, App Services, Service Bus, Notification
